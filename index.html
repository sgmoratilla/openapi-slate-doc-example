    Endpoint conventions

    All services share a common philosophy:

    1. All endpoints follow REST-principles. For an entity, the operations will be POST to create (not repeatable), PUT and PATCH to alter (repeatable), GET to filter entities, a simple GET by the identifier of the entity, DELETE by identifier. For example, for a user, there will be endpoints GET /v2/users, GET PUT PATCH DELETE /v2/users/{user_id}.
    2. Endpoints are nouns and not verbs.
    3. More complex endpoints depend on the service implementation, but they try to follow the principles above.
    4. Not all verbs are implemented for every entity, it depends on the needs of the product, but when adding new endpoints, they follow these principles.
    5. All endpoints must declare explicit parameters. No parameter must be derived from the authenticated user. For instance, if the user_id is important for the request, it must be specified as a param. The main reason is that security is simpler to implement, and it allows to make calls in behalf of a different user.
    6. Same endpoints for different roles. It simplifies the logic, allows us to reuse functionality. On the contrary, it makes security a bit more complex.
    There are exceptions to these rules due to legacy systems, but all new endpoints follow them.

    Pagination

    Not all endpoints support pagination. When they do, they support the parameters ?page={page_index}&size={page_size}. Page indexes start at 0.

    As we chose to return lists in GET endpoints, the total number of possible results has to be returned in a header. That header is total.

    We have detected an incorrect number of results in `total`. It will be fixed as soon as possible.

    Errors

    Anemone endpoints response a custom body when an error is returned. It contains a JSON body with the following fields:

    • status: An error code, dependant of the service, with further explanation of the error.
    • localized_message: A message localized in the locale of the user describing the error.

    localized_message was added to allow clients to give users more feedback when they can't handle the status field. This field is current deprecated or always return in English. The reason is that keeping localization in backend makes it more complicated, forces us to make deploys just for localization. Every time a language is added we have to re-deploy all services.

    {
    "status": "DUPLICATED_EMAIL",
    "localized_message": "This email is already in use. Please choose a different one."
}

    Anemone API uses the standard HTTP error codes:

    Error Code Meaning
    400 Bad Request -- Your request is invalid.
    401 Unauthorized -- Your API key is wrong.
    403 Forbidden -- Your token does not have permission to do that operation.
    404 Not Found -- The specified resource could not be found.
    405 Method Not Allowed -- You tried to access a resiurce with an invalid method.
    500 Internal Server Error -- We had a problem with our server. Try again later.
    503 Service Unavailable -- We're temporarily offline for maintenance or deployment. Please try again later.
